NexusV2 - خارطة الطريق التنظيمية والتطويرية

الدليل الشامل لفصل المشاريع، إدارة سير العمل، وإطلاق الإنتاج

1. فصل المستودعات (Git Repositories)

بسبب التداخل بين الفرونت اند والباك اند والوثائق، الحل الأمثل هو فصلهم لـ 3 مستودعات (Repos)، واستخدام نظام (Git Submodules) لربطهم في مكان واحد.

تحذير هام: هذه الأوامر ستقوم بمسح تتبع Git القديم (ملفات .git) لتنظيف المشروع بالكامل وبدء تتبع جديد مفصول. تأكد من حفظ أي تغييرات غير محفوظة قبل التنفيذ.

خطوات التنفيذ عبر PowerShell:

افتح الـ Terminal في مسار المشروع الرئيسي NexusV2 وقم بنسخ وتشغيل الكود التالي خطوة بخطوة:

# 1. إزالة ملفات Git القديمة لتنظيف المشروع
Remove-Item -Recurse -Force .git -ErrorAction SilentlyContinue
Remove-Item -Recurse -Force .\Nexus-Frontend\.git -ErrorAction SilentlyContinue
Remove-Item -Recurse -Force .\Nexus-backend\.git -ErrorAction SilentlyContinue

# 2. إنشاء مستودع الفرونت اند
cd Nexus-Frontend
git init
git add .
git commit -m "Initial commit for Nexus Frontend"
cd ..

# 3. إنشاء مستودع الباك اند
cd Nexus-backend
git init
git add .
git commit -m "Initial commit for Nexus Backend"
cd ..

# 4. إنشاء مستودع الجذر (Root) للوثائق وربط المشاريع
git init
# إضافة المشاريع الفرعية (عليك إنشاء مستودعات فارغة على GitHub واستبدال الروابط لاحقاً)
# git submodule add <رابط_الفرونت_اند_على_جيت_هاب> Nexus-Frontend
# git submodule add <رابط_الباك_اند_على_جيت_هاب> Nexus-backend

git add NexusV2_Docs Nexus-Docs .gitignore
git commit -m "Initial commit with documentation"

ملاحظة: لتفعيل الـ submodules بشكل صحيح، ستحتاج لإنشاء 3 مستودعات على GitHub أولاً (Backend, Frontend, Root) ورفع الكود لكل واحد منها بشكل منفصل.

2. خطة التطوير والتعامل مع AI Agents

لضمان أن الـ AI Agent يعمل بشكل صحيح بدون أن يخرب المشروع أو يفقد السياق، اتبع هذه القواعد:

كيف تفهمه سياق المشروع؟

  • لا تجعل الـ AI يبحث في المشروع بالكامل. قم بإنشاء ملف باسم .ai-context.md في كل مسار (واحد للباك اند وواحد للفرونت اند) واكتب فيه الهيكلية والقواعد والتقنيات المستخدمة.
  • عند طلب أي تعديل، اطلب منه قراءة ملف ARCHITECTURE_ANALYSIS.md أولاً.

كيف تتأكد من عمله وتنفيذه للمهام؟

  • التجزئة (Micro-Tasks): لا تطلب منه "بناء ميزة الـ Chat". اطلب منه أولاً "تصميم مخطط قاعدة البيانات للـ Chat"، ثم "إنشاء الـ API Endpoint"، ثم "ربط الفرونت اند".
  • طلب الإثبات (Tests): اجعله يكتب Unit Tests أو يشرح لك كيف تقوم باختبار الميزة يدوياً قبل أن تطلب منه إكمال كتابة الكود.
  • تقييد الأوامر: اطلب منه عدم تعديل أي ملف خارج النطاق الذي تحدده له.

كيف تتأكد من تحديثه للوثائق؟

  • اجعل القاعدة رقم 1 في أوامرك: "بعد الانتهاء من الكود والتأكد من عمله، قم بتحديث ملف الوثائق الخاص بهذه الميزة في مجلد NexusV2_Docs".
  • قم بمراجعة ملف الوثائق بعده لتتأكد أنه أضاف الـ Endpoints أو الـ Components الجديدة.

3. سير العمل الصحيح من الـ IDE (VSCode / Cursor)

بما أن المشروع أصبح 3 مستودعات (Repos)، فإن فتح المجلد الرئيسي قد يسبب لك لخبطة. الحل الأفضل هو استخدام Multi-root Workspace.

إعداد بيئة العمل (Workspace)

  • من الـ IDE، قم باختيار File > Add Folder to Workspace وأضف مجلد Nexus-Frontend.
  • كرر العملية وأضف Nexus-backend ومجلد الوثائق NexusV2_Docs.
  • احفظ مساحة العمل: File > Save Workspace As واسمها Nexus.code-workspace.

خطوات الـ Push والـ Merge اليومية

عندما تعمل على ميزة معينة، ستعمل إما على الفرونت اند أو الباك اند أو كلاهما. الخطوات كالتالي:

  1. تحديث الكود محلياً: افتح الـ Terminal الخاص بالقسم الذي تعمل فيه واكتب git pull origin main.
  2. إنشاء فرع جديد: لا تعمل على main مباشرة. اكتب: git checkout -b feature/my-new-feature.
  3. حفظ التعديلات: بعد الانتهاء، من واجهة الـ Source Control في الـ IDE، اضغط على + (Stage) للملفات المعدلة.
  4. كتابة الوصف: اكتب رسالة واضحة (مثال: feat: add user login API) ثم اضغط Commit.
  5. الرفع (Push): اضغط على Publish Branch أو اكتب git push origin feature/my-new-feature.
  6. الدمج (Merge): اذهب إلى GitHub واعمل Pull Request (PR) ثم ادمجه مع الـ main.

4. الدليل الفني للإكتمال (Project Completion Guide)

تم حفظ نسخة Markdown من هذا الدليل في: NexusV2_Docs/project_completion_guide.md

4.1 الهيكل المعماري

  • الفرونت اند (Next.js): يعتمد على مكونات معزولة (Components)، استخدام صارم لـ TypeScript، وإدارة حالة (State Management) مركزية.
  • الباك اند (Laravel/Python): فصل الخدمات إلى Microservices أو وحدات مستقلة (Modular). الاعتماد الكلي على الـ Queues للمهام الثقيلة (مثل WAHA API).

4.2 متطلبات التشغيل والتطوير

  • توحيد بيئات العمل (Environment Variables) وتأمين ملفات .env بحيث لا ترفع أبداً على GitHub.
  • تطبيق أنظمة فحص جودة الكود (Linting & Formatting) أوتوماتيكياً قبل أي Commit.

4.3 متطلبات الإطلاق (Deployment)

  • استخدام منصات مخصصة (مثال: Vercel للفرونت اند، و Laravel Forge أو VPS مخصص للباك اند).
  • تفعيل CI/CD Pipelines عبر GitHub Actions لعمل Tests أوتوماتيكية عند الرفع.
  • تفعيل أدوات المراقبة (Monitoring) مثل Sentry لاصطياد الأخطاء قبل أن يبلغ عنها المستخدم.
