 
# ✅ TasksHub - Master Documentation Suite (v2.0)

## Document 1: FEATURE_REQUIREMENT & BUSINESS LOGIC

### 1.1 Executive Summary
The `TasksHub` is the centralized execution and management layer for all actionable items within the Nexus platform. It manages the lifecycle of tasks whether they are to be performed manually by a human (Hedra), programmatically via code execution, or autonomously by an AI Agent. It bridges the gap between static to-do lists and autonomous AI workforce orchestration.

### 1.2 Task Classifications & Execution Modes
*   **Manual Tasks:** Traditional to-do items requiring human intervention (e.g., "Call client X to finalize contract").
*   **Agentic Tasks:** Tasks delegated to a specific AI Agent from the `AgentsHub` (e.g., "Draft a proposal based on the latest conversation with Contact Y").
*   **System/Code Tasks:** Automated scripts or API calls executed by the backend.

### 1.3 Trigger & Scheduling Capabilities
*   **On-Demand:** Created and triggered manually via the UI.
*   **Scheduled (Recurring/One-off):** Tasks integrated with the `SchedulerHub` using Cron expressions (e.g., "Generate weekly summary report every Sunday at 9 AM").
*   **Workflow-Triggered:** Tasks generated dynamically as a step within a running workflow from the `WorkflowsHub`.

### 1.4 Task Lifecycle & Statuses
Tasks follow a strict state machine:
*   `Todo`: Created but not yet started.
*   `In-Progress`: Actively being worked on by an Agent, system, or human.
*   `Blocked`: Paused due to missing dependencies, human approval requirements, or external API failures.
*   `Completed`: Successfully finished.
*   `Failed`: Unrecoverable error occurred during execution.
*   `Cancelled`: Aborted before completion.

### 1.5 Contextual Metadata
Every task can be heavily contextualized. A task can be linked to:
*   A specific **Contact** (from `ContactsHub`).
*   A specific **Conversation** (from `PeopleConnect`).
*   Specific **Memory Chunks** (providing the Agent with the exact background needed to execute the task).

## Document 2: ARCHITECTURE & BACKEND SPECIFICATION (Laravel 11)

### 2.1 Database Schema (MySQL)
*   **`tasks` Table:**
    *   `id` (UUID, Primary)
    *   `title` (VARCHAR)
    *   `description` (TEXT)
    *   `type` (VARCHAR) - manual, agent, system.
    *   `status` (VARCHAR) - todo, in-progress, blocked, completed, failed, cancelled.
    *   `priority` (VARCHAR) - low, medium, high, critical.
    *   `assigned_agent_id` (UUID, Foreign Key, Nullable)
    *   `contact_id` (UUID, Foreign Key, Nullable)
    *   `workflow_execution_id` (UUID, Foreign Key, Nullable)
    *   `due_date` (DATETIME, Nullable)
    *   `payload_data` (JSON) - Configuration or context passed to the Agent/System.
    *   `result_data` (JSON) - The final output generated by the task.
*   **`task_logs` Table:** (Integrated with LogsHub)
    *   `id` (UUID)
    *   `task_id` (UUID)
    *   `message` (TEXT)
    *   `level` (VARCHAR)
    *   `created_at` (TIMESTAMP)

### 2.2 Core Service Layer (Strict SRP)
*   **`TaskManagementService`:** Handles CRUD operations, state transitions, and validation.
*   **`TaskExecutionService`:** The dispatcher. If `type == 'agent'`, it packages the `payload_data`, resolves the assigned agent, and dispatches an asynchronous job (`ExecuteAgentTaskJob`) to the Redis queue.
*   **`TaskSchedulingService`:** Integrates with Laravel's Task Scheduler (Cron) to evaluate recurring tasks and push due tasks into the `Todo` or `In-Progress` queues.

### 2.3 Background Jobs & Queues (Redis/Horizon)
*   Nexus enforces a **Background-First** architecture. When a task is marked for execution, it is pushed to Redis.
*   `ExecuteAgentTaskJob`: Handles the heavy lifting, calling the `AgentsHub`, waiting for the LLM response, updating the task to `Completed`, and saving the output in `result_data`.
*   If the job fails (e.g., API timeout), it utilizes Laravel's exponential backoff retries before updating the status to `Failed` and pushing it to the Dead Letter Queue (DLQ).

### 2.4 API Contract
*   `GET /api/v1/tasks` : List tasks (Filters: status, priority, due_date, agent_id).
*   `POST /api/v1/tasks` : Create a new task.
*   `PATCH /api/v1/tasks/{id}/status` : Update status (Triggers state machine logic).
*   `POST /api/v1/tasks/{id}/execute` : Manually force execution of an agent/system task.
*   `GET /api/v1/tasks/{id}/logs` : Retrieve execution logs for a specific task.

## Document 3: UI_DESIGN_LAYOUT & FRONTEND UX (Next.js 15)

### 3.1 Hub Layout & Navigation
The `TasksHub` provides two primary visualization modes:
1.  **Kanban Board View:** The default view. Columns representing statuses (`Todo`, `In-Progress`, `Blocked`, `Completed`). Powered by `NxDragDropZone` for intuitive status updates.
2.  **List/Table View:** A dense `NxDataGrid` for power-user bulk operations, sorting, and advanced filtering.

### 3.2 UI Components
*   **`NxTaskCard`:** Used in the Kanban view. Displays Title, Priority Badge (`NxStatusBadge`), Assigned Agent Avatar, Due Date, and Contact Link.
*   **`NxTaskModal`:** A comprehensive slide-over or modal for creating/editing a task. Contains:
    *   `NxInput` for Title/Description.
    *   `NxSelect` for Priority, Status, and Agent Assignment.
    *   `NxDateTimePicker` for scheduling.
    *   **Context Attacher:** A sub-component to link specific Contacts or Conversations.
*   **`NxTaskExecutionLog`:** A read-only terminal-like component inside the Task Details view showing real-time Reverb WebSocket updates as an Agent processes the task.

### 3.3 State Management (Zustand: `useTasksStore`)
*   **State:** `tasks`, `filters`, `activeView` (kanban/list), `isExecuting`.
*   **Actions:** `fetchTasks()`, `createTask()`, `updateTaskStatus(id, newStatus)`.
*   **Optimistic Updates:** When a user drags an `NxTaskCard` from `Todo` to `Completed`, the UI updates instantly while the Axios patch request runs in the background. If the request fails, the card snaps back to its original column.

## Document 4: INTEGRATION, SECURITY & OPERATIONS

### 4.1 Cross-Hub Integrations
*   **WorkflowsHub:** A Workflow can contain a step of type `CreateTask`. The workflow creates the task in `TasksHub` and enters a `Paused` state. Once the task reaches `Completed`, the `TasksHub` fires an event (`TaskCompletedEvent`) that wakes the workflow up to continue to the next step.
*   **AgentsHub:** Tightly coupled for execution. The `TasksHub` acts as the queue and logging interface, while the `AgentsHub` provides the cognitive persona and tools to actually do the work.
*   **LogsHub:** Every state transition (e.g., Todo -> In-Progress) and execution step generates a log entry sent to the `LogsHub` via polymorphic relations (`related_type: Task`, `related_id: UUID`).

### 4.2 Security & Operational Resilience
*   **Rate Limiting & Concurrency:** Agentic tasks are pushed to a specific Horizon queue (`agent-tasks`) with a strict concurrency limit to prevent overwhelming the `AIModelsHub` and triggering 429 Too Many Requests errors.
*   **Dead Letter Queue (DLQ) Management:** Failed tasks can be manually reviewed in the `TasksHub` UI. The user can view the `error_message` in the logs, fix the context, and click a "Retry" `NxActionButton`.
*   **Data Integrity:** Task deletions are implemented as **Soft Deletes** in MySQL to ensure historical execution logs and Workflow audits are never orphaned.
